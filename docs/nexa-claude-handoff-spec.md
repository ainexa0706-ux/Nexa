# Nexa 完全引き継ぎ設計書

> この文書は、AI開発者が既存の Nexa を理解し、既存機能を壊さずに改善・再現するための唯一の引き継ぎ資料です。実装前に必ずリポジトリを読み、実画面を起動して確認してください。新しいアプリをゼロから作ることは目的ではありません。

## 1. プロダクトの目的

Nexa は、通常の会話AIとローカル開発エージェントを一つのプロジェクト型ワークスペースに統合するWindowsデスクトップアプリです。

- ユーザーは会話、相談、文章作成、画像生成を素早く行える。
- コード作業ではユーザーが選択したPC上のフォルダーを直接扱い、ファイル作成・編集・削除・コマンド実行・検証を継続できる。
- 会話履歴は単なるスレッド一覧ではなく、**プロジェクト、選択フォルダー、最終目的、記憶、実行履歴**と一緒に保存される。
- 短い指示（例: 「続き」「動くように」「もう一回」）でも、直前の会話、最終目的、実際のワークスペースを根拠に意図を復元して作業を続ける。
- 見た目は高密度で上質な開発ワークスペース。会話だけの時は静かで分かりやすく、コード作業中だけ実行状況と変更内容を明確に見せる。

Nexa は特定の既存製品のコピーではない。目標は「会話の自然さ」と「ツールを使う開発エージェントの確実さ」を独自の名前と体験で統合すること。

## 2. 絶対に守る原則

1. 既存の Node.js + Electron + ローカルWeb UI 構成を維持する。大規模なフレームワーク置換やゼロからの再構築は禁止。
2. 動いているAPI、認証、課金、保存済みプロジェクト、デスクトップ配布を壊さない。
3. キーワードごとの固定応答を増やして問題を隠さない。会話文脈・最終目的・ワークスペース状態から意図を判断する。
4. 会話モードとコードモードは、**理解能力・会話履歴・関連記憶・最終目的の使い方を同じ品質にする**。違いは許可されたツールと画面だけ。
5. コードモードで「説明だけ出して保存に失敗」を成功扱いしない。実ファイル変更と検証結果を根拠に完了を表示する。
6. ファイル操作は選択フォルダーを基準にする。チャットごとのダウンロードや別の自動生成フォルダーを作らない。
7. 開発ログは内部推論の全文ではなく、ユーザーが追える具体的な作業の実況にする。
8. 実際に動かしたものだけを「完了」「生成済み」「検証済み」と表示する。動画・画像・外部検索などについて作ったふりをしない。
9. 変更後は必ず lint、test、build、実画面確認を行う。型回避のための `any`、ダミーUI、クリックできないボタンは禁止。

## 5. モード仕様

### チャットモード

- 目的: 会話、相談、説明、文章作成、画像生成。
- フォルダー選択は不要。
- コードの直接保存、端末操作、変更ファイルカード、開発ログ、AIチームの技術説明は表示しない。
- 会話は逐次ストリーミングする。内部の英語思考文、メタ会話、プロンプト断片を本文へ漏らさない。
- 質問が必要な時は本文の末尾で曖昧に質問せず、選択カードで選べるようにする。
- 画像生成はアプリ内プレビュー。実際の生成物がある時のみ表示する。

### コードモード

- 最初にユーザーのPC上の作業フォルダーを選択する。
- 以後のチャットでは同じプロジェクトとフォルダーを維持する。モードを途中で勝手に解除しない。
- 同じ理解モデル、会話履歴、最終目的、関連記憶を使ってから、構造化されたファイル操作に変換する。
- 作業単位は `list_files`、`read_file`、`write_file`、`delete_file`、`run_command`、`finish`。
- 既存ファイルを上書きする前に読む。パスは必ず選択フォルダーからの相対パス。
- 任意の数のファイルを扱える。3ファイル制限は存在しない。
- 実行結果は会話下の変更カードと右パネルの出力へ連動する。

### 両方モード

- チャットとコード作業を併用する。
- フォルダーを選択していればコード操作を有効にし、未選択なら通常会話として開始できる。
- 右側にはコード変更、生成結果、ファイル、AIチーム、出力ログをモードに応じて表示する。

## 7. 意図理解・エージェント方式

### 必須フロー

```text
ユーザー入力
  -> 意味・文脈判断
  -> 最終目的 / 直近会話 / 関連記憶 / ワークスペース確認
  -> 要求を明文化した execution request
  -> モード・安全性・ツールの選択
  -> 実装または会話生成
  -> 批評・検証・品質判定
  -> 必要なら限定的な再生成
  -> 保存・実況・最終回答
```

### 内部役割

UIには役割名を日本語で表示する。ユーザー向け本文で外部AI会社名や他製品名を名乗らない。

- 司令塔: 全体の優先順位と最終回答の責任。
- 計画: 要件、成功条件、影響範囲、実装順序。
- 記憶: プロジェクト記憶と会話文脈から関連情報を取り出す。
- ツール: ファイル、端末、添付、生成物を適切な範囲で選ぶ。
- 推論: 短い依頼や曖昧な続きの意図復元。
- 設計: 構成、データフロー、実行方法、テスト範囲。
- コード: 実ファイルの作成・編集・削除。
- 調査: 必要な場合のみ。検索機能が無効なら現在の情報だと偽らない。
- 別視点: 抜け、古い文脈への流れ、過大表現を確認。
- 批評 / 検証: 要求とのズレ、構文、起動、テスト、危険性を確認。
- 品質: 最終回答の採点と必要な補正。

### 実装上の重要点

1. 意図判断はキーワード辞書の追加だけに頼らない。`resolveTurnIntentWithAi` のように、会話、workspace、最終目的、直前の実装を入力として構造化JSONを返す。
2. `resolvedRequest` はユーザーへの表示用の説明で終わらせず、実装エージェントの primary request に渡す。
3. コードモードの主モデルは会話モードと同じ intelligence model を優先する。コード専用モデルはJSONツール出力やフォールバックに使えても、理解の質を落とす理由にしない。
4. structured code agent は1ターン1操作JSONのみ返す。本文のファイルフェンスを解析して保存する旧方式に依存しない。
5. JSONが壊れた時は再試行の理由と直前のツール結果を渡す。空のテンプレートへ勝手に置換しない。

## 8. 構造化コードエージェント

### 操作契約

```json
{"action":"list_files","reason":"..."}
{"action":"read_file","path":"relative/path","reason":"..."}
{"action":"write_file","path":"relative/path","content":"complete content","reason":"..."}
{"action":"delete_file","path":"relative/path","reason":"..."}
{"action":"run_command","command":"project-scoped command","reason":"..."}
{"action":"finish","summary":"implemented and verified"}
```

### 完了条件

- 少なくとも必要なファイル変更がある。
- 実際のワークスペースでファイル存在と最低限の構文を確認する。
- `package.json` がある時は既存の lint/typecheck/test/build スクリプトを尊重する。
- 失敗時は該当ファイルを読んで修正し、もう一度検証する。
- 要求された主要動作、依存関係、エントリーポイント、実行説明が不足していない。
- 検証不合格の変更を「完了」として残さない。必要なら変更を安全に戻す。

### 3D・大規模開発の判断

- フォルダー名が `UnityProject` であるだけではUnityプロジェクトと判断しない。`Packages/manifest.json` と `ProjectSettings` など実際のメタデータを確認する。
- 空の3Dゲーム依頼は、明示的なエンジン指定がなければ一貫した実行可能なWeb 3Dスタック（例: Vite + Three.js）を選ぶ。
- 大規模アプリでは、単一HTMLの玩具に縮小しない。画面、状態、永続化、エラー状態、テスト、実行手順を適切に分割する。
- 「リアル」「高品質」「本格的」はデモや最小構成に言い換えない。ただし素材、ネットワーク対戦、商用アセットなど実現できないものは正直に段階を示す。

## 9. アクセス権と安全性

| 権限 | 挙動 |
| --- | --- |
| 読み取り専用 | ファイル参照・計画・差分プレビューのみ。書き込み・端末実行は不可。 |
| 承認を常に求める | 外部ファイル編集、端末、インターネット利用前に毎回確認。 |
| 危険時に承認 | 通常の選択フォルダー作業は進め、削除、秘密情報、外部送信、破壊的操作だけ確認。 |
| フルアクセス | ユーザーが明示的に選んだ時のみ、不要な確認を挟まず作業する。ただしOS破壊、資格情報露出、外部送信などは内部安全策を維持する。 |

削除依頼では、対象フォルダー、保護対象（`.git`、`.env`、ユーザーデータ等）、退避またはバックアップ、最終確認を明確にする。ユーザーが「実行せず」と言った時は、絶対に書き込みや端末実行をしない。

## 10. UI/UX仕様

### 全体レイアウト

```text
左: プロジェクト・チャット履歴
中央: ヘッダー / 実行設定 / 会話 / 開発実況 / コンポーザー
右: 最終目的 / 出力 / 情報源 / AIチーム / モデル性能 / メモリー
```

- デスクトップは固定幅の左右サイドバーと `minmax(0, 1fr)` の中央列。横スクロールを出さない。
- 1366px、1440px、1920pxで崩れない。狭い画面とモバイルではサイドバーを折りたためる。
- 新規チャットかつメッセージ0件なら中央に大きい入力欄とモード選択を表示する。開始後は入力欄を下部へ移す。
- ダークテーマ、控えめな境界線、8px前後の角丸、読みやすい高コントラスト。装飾だけのグラデーションや過度なカード入れ子を避ける。
- 左下には課金プラン入口がある。ログインフォームを左下へ混ぜない。

### モード別表示

- チャット: 会話本文と生成結果中心。開発実況、変更カード、技術的なAIチーム説明を隠す。
- コード: 下部に進行ステップ（要件分析、設計、実装、テスト、ドキュメント）、会話内に実況と変更カードを出す。
- 両方: 必要なパネルのみ表示し、未選択フォルダー時はコード変更操作を無効状態にする。

### ライブ進行（重要）

開発ログは単なる箇条書きでも、モデルの思考全文でもない。進行中は会話の直下に以下を表示する。

```text
Nexaが実行中
現在: src/game/player.js を編集しました
入力処理と移動ロジックを追加しています。変更行: 84

いま行っている作業と履歴  [+/-]
  作業を開始
  作業フォルダーを調査
  package.json を確認しました
  src/game/player.js を編集しました
  npm run build を実行
  最後の確認が完了しました
```

- 表示対象: 開始、意図決定、フォルダー確認、既存ファイル確認、実ファイルの作成/編集/削除、コマンド、検証、エラー、完了。
- 非表示対象: 長い内部推論、プロンプトの言い換え、AI同士の無意味な会話、抽象的な「考えています」の連続。
- 最新イベントを固定して表示し、履歴は `details` で展開・最小化できる。
- ユーザーが上へスクロール中は回答中でも強制的に最下部へ戻さない。

### 変更カード

ファイル変更があった最終回答の直下に1つの変更カードを出す。

- 変更したファイル一覧、追加/削除行数、選択中ファイルの差分、レビュー、ファイルを開く操作を同一カードに置く。
- ファイルをクリックすると差分プレビューを切り替える。
- 「適用済み」は実際に選択フォルダーへ保存できた時だけ表示する。
- 古い別ログに埋もれさせない。

## 11. 画像・動画・添付

- コンポーザー左の `+` は添付、アクセス権変更、フォルダー選択、プランモードなど拡張操作の土台。
- 添付はOSのファイル選択と同じ体験にし、添付内容を会話の根拠として利用する。
- 画像生成はアプリ内カードでプレビューする。外部ブラウザーへ飛ばさない。
- 動画生成は実際のプロバイダーと生成物が利用できる時だけ動画プレビューを出す。テキストだけを「動画生成済み」と言わない。
- 現在のビルドで動画が無効なら、画像または絵コンテ代替を正直に提示する。

## 12. 認証、課金、管理

- 本物のメール認証とGoogle OAuthを分離して扱う。Googleのデスクトップ認証はElectronのリダイレクトURLとGoogle Cloud設定を一致させる。
- アプリを閉じてもログインを維持する。セッションは `%APPDATA%\Nexa` の安全な保存領域を使う。
- 認証画面、課金画面、API管理画面を同一画面に混ぜない。
- 課金は独立した画面/導線にし、Stripeキーを設定するだけで本番接続できる準備を維持する。
- 管理者は指定アカウントだけ。一般公開の管理者LPは作らない。管理者はユーザー一覧、設定、BANを扱える。
- API公開は将来的な機能であり、現在のローカルNexaを勝手に外部課金APIへ変えない。

## 13. LP、公開、配布

- LPはアプリの実機能、コード作業、プロジェクト継続、ライブ進行、料金を紹介する。画像生成機能をLPの主役にしない。
- LPからWindowsインストーラーへ直接到達できるようにする。
- Electron NSISインストーラーを使う。`npm run dist:win` が配布物を生成する。
- バージョンはリリース規則に沿って上げる。更新時に既存ユーザーのデータを消さない。
- 公開URL、LP URL、ダウンロードURLは実際に到達可能なものだけ案内する。

## 14. 既知の失敗と再発防止

| 失敗 | 原因 | 正しい修正 |
| --- | --- | --- |
| 「動くようにして」で勝手に新しい汎用Webアプリを作る | 短文をキーワードで新規作成へ分類 | 最終目的、直前の実装、実ワークスペースを見て debug/continue を判断する。 |
| 「もう一回」で目的が失われる | 続き文を新しい最終目的として保存 | 最終目的は固定。短文は既存目的の続行指示として扱う。 |
| ファイル保存が `writable_file_block_or_valid_diff_not_found` で終わる | 自然言語ファイルブロック出力に依存 | 構造化JSONの1操作ずつのツールエージェントを使う。 |
| 生成が何度も再試行してテンプレートに置換される | 失敗原因と実ワークスペースを引き継がない | 直前の読み取り結果、変更済みファイル、検証結果を次アクションへ渡す。 |
| 3DゲームがUnity/C#/Python/Webを行き来する | フォルダー名や既存出力を根拠なく採用 | 実際のメタデータを調査し、1つの一貫したスタックを選ぶ。 |
| 会話モードで内部英語の思考が出る | モデル出力をそのまま表示 | ストリームフィルター、内部推論検出、最終回答再生成を使う。 |
| コードモードの理解力が低い | 会話モデルとコードモデルで文脈経路が分断 | 同じ conversation context / final objective / memory / primary modelを先に使い、最後だけツール操作に変換する。 |
| 作業ログが何をしているか分からない | 計画の列挙だけを表示 | 実際のファイル、コマンド、エラー、検証の実況を最新状態として表示する。 |
| 会話履歴が消えたように見える | 開発用 `data` とElectronの `%APPDATA%` データを混同 | 保存先を明示し、移行・削除時は両方のindexとproject JSONを確認する。 |

## 15. 実装チェックリスト

### コード変更前

1. `rg --files` と `rg` で既存構造・実装・テストを読む。
2. `git status --short` を確認し、ユーザーの未コミット変更を戻さない。
3. アプリを起動し、対象画面を実際に確認する。
4. 既存データ保存先と対象フォルダーを確認する。

### コード変更後

```powershell
npm run lint
npm test
npm run build
npm run dev
```

## 18. AI開発者への最終指示

1. この文書、`README.md`、`server.mjs`、`public/app.js`、`public/styles.css`、`electron/main.cjs` を読んでから実装する。
2. 画面だけを似せず、プロジェクト永続化、最終目的、意図復元、構造化ファイル操作、検証、実況を一つの動作として実装する。
3. バグを見つけた時は、その症状だけのキーワード条件を足すのではなく、データフロー、保存状態、モード分岐、モデル選択、ツール結果の受け渡しを調査して根本原因を直す。
4. ユーザーに見せる文面は日本語で自然に、短く、正直にする。内部推論、モデルの独り言、他社製品名、未実行の結果を混ぜない。
5. 変更を終える前に、実画面、主要クリック操作、保存後の再起動、lint/test/buildを確認し、修正内容と残る制約を報告する。
